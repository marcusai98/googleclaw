Kernprincipe

OpenClaw IS de backend

Het verschil met "verkapte software" zoals Empira: bij Empira draait er een onafhankelijke applicatie die toevallig AI gebruikt. Bij GoogleClaw IS OpenClaw de runtime. Er bestaat geen GoogleClaw zonder OpenClaw — het is geen aparte software, het is een OpenClaw-configuratie met agents, files en een web-interface.

❌
Verkapte software (Empira-stijl)
Eigen backend, eigen database, eigen auth, eigen agent runtime. OpenClaw is een tool erin. Je zou OpenClaw kunnen vervangen.
✅
OpenClaw native (GoogleClaw)
OpenClaw is de runtime, het geheugen, de executor. De web app is een scherm dat via de gateway communiceert. Zonder OpenClaw werkt er niets.
❌
Losse Python agents
Scripts die zelfstandig draaien, eigen cron daemon, eigen logging, eigen API keys beheren buiten OpenClaw om.
✅
OpenClaw sub-agent sessies
Agents leven als OpenClaw sessies. Crons spawnen sub-agents. Memory in workspace files. Tools via OpenClaw's eigen tool-ecosysteem.

De sleutel

OpenClaw Gateway — /tools/invoke

OpenClaw's gateway exposed standaard een HTTP endpoint op hetzelfde poort als de WebSocket. De web app roept dit aan als een REST API — met de gateway token als auth.

# Data lezen (web app → OpenClaw → workspace file)
POST http://[vps-ip]:63783/tools/invoke
Authorization: Bearer [gateway-token]
Content-Type: application/json

{
  "tool": "read",
  "args": { "path": "/opt/googleclaw/workspace/queue.json" }
}

# Agent aansturen — approval sturen naar main sessie
{
  "tool": "sessions_send",
  "args": {
    "sessionKey": "main",
    "message": "APPROVE scale:led-desk-lamp-store"
  }
}
sessions_send zit standaard op de gateway deny list. Één config-wijziging op de VPS unlock dit:

{ "gateway": { "tools": { "allow": ["sessions_send"] } } }

Wat de web app via /tools/invoke kan

Dashboard data ladenread{ path: "dashboard.json" }
Agent status ladenread{ path: "agents-status.json" }
Item goedkeurensessions_send{ sessionKey: "main", message: "APPROVE ..." }

Caddy als CORS-oplossing

De web app en de gateway zitten op verschillende origins. Caddy op de VPS serveert de web app en proxiet /tools/invoke — zo is alles dezelfde origin, geen CORS issues.

googleclaw.icarusagency.io {
  # Serveert mobile.html
  handle /mobile.html {
    root * /opt/googleclaw/frontend
    file_server
  }

  # Proxiet gateway calls (CORS-vrij, zelfde origin)
  handle /tools/invoke {
    reverse_proxy 127.0.0.1:63783
  }
}
State management

Workspace files als database

Geen externe database. OpenClaw agents schrijven JSON files naar de workspace. De web app leest ze via de gateway. Simpel, transparant, volledig binnen OpenClaw's eigen systeem.

Telegram werkt parallel. Henk kan ook gewoon via Telegram "approve scale led-store" sturen. OpenClaw verwerkt het identiek — dezelfde agent, dezelfde tools, hetzelfde resultaat. De web app is een extra kanaal, geen vervanging.
Agent architectuur

5 agents — allemaal OpenClaw sub-agent sessies

Elke agent is een OpenClaw cron die een sub-agent sessie spawnt. Die sessie heeft toegang tot alle OpenClaw tools, schrijft naar workspace files, en leest zijn eigen memory. Geen aparte processen, geen eigen runtime.

🧠
HERALD
Performance Monitor — meest waardevolle agent
Cron
0 6 * * * — dagelijks 06:00
Sessie task
Read agents/herald/state.md → query Google Ads API voor elke actieve store → schrijf scale/alert items naar queue.json → update agents/herald/state.md
Tools gebruikt
exec (google_ads script) read (state.md) write (queue.json) message (Telegram notify)
Memory
agents/herald/state.md — last run, scale success rates per store, per-category ROAS baselines
Leert
Na elke approved scale: was ROAS 7 dagen later nog boven target? Update success rate. Pas budget-increment aan per store-categorie.
🔍
SCOUT
Product Research — meest complexe agent
Cron
0 2 * * * — dagelijks 02:00
Sessie task
Haal trending producten op via Google Trends + AliExpress → score elk product 0-100 → schrijf top items naar queue.json → update state.md
Tools gebruikt
web_search (trends) web_fetch (AliExpress) exec (pytrends script) write (queue.json) image (product analyse)
Scoring
trend (30pt) + marge (25pt) + concurrentie (20pt) + leverancier (15pt) + timing (10pt) = 0–100. Score ≥80 auto-approve, 60–79 naar Inbox, <60 discard.
Leert
Elke approval/rejection gelogd in agents/scout/outcomes.json. Wekelijks: herbereken scoring-gewichten op basis van 30-daags resultaat.
💰
MIDAS
Finance — simpelste agent, bouw eerst
Cron
0 7 * * * — dagelijks 07:00
Sessie task
Haal revenue op via MariaDB v_store_performance + spend via Google Ads API → schrijf dashboard.json → update agents-status.json
Tools gebruikt
exec (mariadb query) exec (google ads script) write (dashboard.json)
Output
Schrijft dashboard.json met 7d revenue/ROAS/profit arrays. Geen approvals nodig — read-only agent.
🏪
LISTER
Store & Listings — event-driven
Trigger
OpenClaw main sessie ontvangt "APPROVE product:xyz" → spawnt LISTER sub-agent met product payload
Sessie task
Genereer productafbeeldingen → schrijf titel/beschrijving → push naar Shopify API → update queue.json status → notificeer Henk
Tools gebruikt
exec (Gemini image gen) exec (shopify API script) write (queue.json update) message (Telegram)
📊
COMMANDER
Google Ads Manager — execute only after approval
Cron
0 5 * * 1 — maandag 05:00
Sessie task
Haal search terms report op (14 dagen) → vind wasted spend → schrijf optimize items naar queue.json
Na approval
OpenClaw ontvangt "APPROVE optimize:xyz" → spawnt COMMANDER execution sub-agent → voegt negatives toe via Google Ads API op campagne-niveau
Leert
Schrijft waste-patronen naar agents/commander/waste-patterns.md. Opbouw van universele waste-lijst per product-categorie.
Self-learning

Agents leren van elke beslissing

Elke approval of rejection wordt opgeslagen in de agent's eigen memory files. Wekelijks draait een synthesis sub-agent die de patronen analyseert en de agent-configuratie bijwerkt.

Memory per agent — twee lagen

LaagLocatieInhoud
Structureelagents/[naam]/state.mdLast run, thresholds, succes-percentages
Narratiefagents/[naam]/learnings.mdMenselijk leesbare patronen die agent geleerd heeft
Uitkomstenagents/scout/outcomes.jsonElke product-beslissing met 30d resultaat

Weekly synthesis — zondag 20:00

SYNTHESIS sub-agent wordt gespawnt:
1. Lees queue.json — alle beslissingen van afgelopen week
2. Bereken per agent: approval rate, succes rate, gemiddeld resultaat
3. Update state.md per agent met nieuwe thresholds
4. Schrijf samenvatting naar agents/[naam]/learnings.md
5. Als kritieke regel wijzigt → stuur notificatie naar Henk

Build volgorde

10 stappen — van lege VPS naar V1 live

Gateway config op VPS
sessions_send unblocken in openclaw.json. Gateway token veiligstellen. Testen met curl.
Workspace structuur aanmaken
Alle mappen en lege JSON files: queue.json, dashboard.json, agents-status.json, agents/*/state.md
Caddy configureren op VPS
mobile.html serveren + /tools/invoke proxyen naar gateway. HTTPS via auto-cert. Testen via browser.
Web app koppelen aan gateway
mobile.html fetch() calls vervangen — queue.json en dashboard.json via /tools/invoke lezen. Approve buttons sturen sessions_send.
MIDAS agent bouwen
OpenClaw cron → sub-agent sessie → MariaDB + Google Ads query → schrijft echte dashboard.json. Web app toont live data.
HERALD agent bouwen
Meest waardevolle agent. Cron → sub-agent → echte scale/alert items in queue.json → Henk goedkeurt in web app.
Approval execution in main sessie
Main sessie verwerkt "APPROVE scale:xyz" bericht → voert actie uit via Google Ads tools → update queue.json status.
SCOUT agent bouwen
Meest complex. Trends + AliExpress + scoring model. Product cards in queue.json.
LISTER agent bouwen
Triggered door product approval. Afbeeldingen genereren + Shopify push.
COMMANDER + weekly synthesis
Keyword audit + self-learning loop. Na V1 live, zodra er approval-data is om van te leren.
